# Stology 온톨로지 정의

## 1. 한 줄 정의

Stology의 온톨로지는 사용자가 직접 복잡한 지식 그래프를 만드는 도구가 아니라, **서비스가 미리 제공하는 개념 지도 템플릿**이다.

스터디는 이 템플릿을 선택해 시작하고, 업로드한 학습자료가 템플릿 안의 개념 노드와 연결되면서 스터디별 지식 구조가 채워진다.

```text
서비스 제공 온톨로지 = 표준 개념 지도
스터디 학습자료 = 개념을 활성화하는 근거
스터디 지식 구조 = 표준 개념 지도 + 우리 스터디의 진행 상태
```

---

## 2. 왜 온톨로지를 제공하는가

스터디에서는 매주 발표자료, 정리글, 질문, 실습 기록 같은 자료가 쌓인다. 하지만 자료가 많아질수록 다음 문제가 생긴다.

- 어떤 개념을 이미 다뤘는지 한눈에 보기 어렵다.
- 새로 배운 개념이 기존 개념과 어떻게 연결되는지 파악하기 어렵다.
- AI가 추출한 개념을 그대로 믿으면 잘못된 지식 구조가 쌓일 수 있다.
- 스터디장이 매주 부족 개념과 다음 학습 방향을 직접 판단해야 한다.

Stology는 이 문제를 해결하기 위해 **미리 정의된 공통 온톨로지**를 제공한다. 사용자는 그래프 편집 지식이 없어도, 자료 업로드와 팀 검토만으로 스터디의 지식 구조를 쌓을 수 있다.

---

## 3. 서비스에서 제공하는 온톨로지의 형식

### 3.1 온톨로지 팩

Stology는 학습 분야별로 **온톨로지 팩**을 제공한다.

```text
1. Spring / Spring Boot 백엔드 스터디 온톨로지 팩
2. React 프론트엔드 스터디 온톨로지 팩
...
```

### 3.2 기본 단위: Concept

온톨로지의 노드는 모두 **Concept**다.

Concept는 학습자가 이해해야 하는 개념을 뜻한다.

예시:

```text
Spring
Spring Boot
DI
Bean
Controller
Service
Repository
JPA
JWT
```

자료, 질문, 답변, 발표 파일은 온톨로지 노드가 아니다. 이들은 Concept를 활성화하거나 설명하는 **근거 자료**로 연결된다.

### 3.3 관계 형식

Stology는 개념 사이의 관계를 사용자가 이해하기 쉬운 4가지로 제한한다.

| 관계명 | JSON key | 의미 | 예시 |
| --- | --- | --- | --- |
| 기반 개념 | `based-on` | 어떤 개념을 이해하기 위해 먼저 알아야 하는 개념 | Bean은 DI를 기반 개념으로 가짐 |
| 맥락 개념 | `associated-with` | 같은 구조나 흐름 안에서 함께 이해되는 개념 | Controller, Service, Repository |
| 확장 개념 | `advanced-from` | 더 깊은 학습이나 응용으로 이어지는 개념 | JWT는 Spring Security에서 확장 |
| 대조 개념 | `contrasted-with` | 비슷하지만 구분해야 하는 개념 | Authentication과 Authorization |

### 3.4 주차 형식

온톨로지 템플릿의 각 Concept는 `week` 값을 하나만 가진다.

`week`는 해당 개념이 자료에서 처음 본격적으로 학습되거나 이후 학습의 전제로 다뤄지는 주차를 뜻한다. 단순 언급, 예고, “추후에 배울 내용” 수준의 참조는 해당 주차로 보지 않는다.

## 4. 스터디에서 온톨로지가 작동하는 방식

### 4.1 스터디 생성

스터디장은 스터디 생성 시 서비스가 제공하는 온톨로지 템플릿을 선택한다.

```text
스터디 이름 입력
→ 온톨로지 템플릿 선택
→ 시작일 입력
→ 스터디 생성
```

스터디 생성 직후 지식 구조 탭에는 선택한 온톨로지 팩의 전체 노드가 표시된다. 아직 자료가 연결되지 않은 노드는 회색 반투명 상태다.

### 4.2 자료 업로드

스터디원은 마크다운 파일을 업로드하거나 텍스트를 직접 입력한다.

AI는 자료에서 핵심 개념 후보를 추출하고, 제공 온톨로지 팩에 있는 Concept와 매칭한다.

```text
학습자료
→ AI 개념 후보 추출
→ 제공 온톨로지 Concept와 매칭
→ 검토 화면으로 전달
```

### 4.3 팀 검토

AI 결과는 바로 확정되지 않는다.

스터디원은 GitHub PR 방식과 비슷하게 노드 후보별로 **승인 / 반려**를 선택한다.

### 4.4 지식 구조 갱신

검토를 통과한 Concept는 지식 구조 탭에서 활성화된다.

```text
회색 반투명 노드
→ 승인된 자료와 연결
→ 활성 노드로 표시
→ 관련 자료와 연결 관계 확인 가능
```

### 4.5 주차별 리포트 연결

활성화된 Concept와 자료 기록은 주차별 리포트에 활용된다.

리포트의 `이번 주 핵심 노드`는 활동 결과가 아니라 온톨로지 템플릿의 주차 메타데이터를 기준으로 정한다.

```text
4주차 리포트
→ 온톨로지 템플릿에서 Concept.week = 4 조회
→ 활성/비활성 및 자료 연결 여부와 무관하게 목록에 포함
→ 노드명, 현재 학습 상태, 관련 자료 수 표시
```

`week = 0`인 공통·선행 개념은 특정 주차의 핵심 노드 목록에서 제외한다. 신규 활성화/보강 노드는 해당 주차에 실제로 발생한 활동 결과이므로, 주차 메타데이터 기반 핵심 노드와 서로 다른 집합일 수 있다.

리포트는 다음 정보를 보여준다.

- 이번 주 활성화 노드 수
- 신규 활성화 노드와 보강 노드
- 이번 주 학습 내용 요약
- 이번 주 핵심 노드
- 추천 노드와 개별 추천 사유
- 팀원 활동

즉, 온톨로지는 단순한 그래프 화면이 아니라 **스터디 운영 판단과 다음 자료 준비를 돕는 기준 구조**다.

## 5. 예시

### 5.1 제공 온톨로지 일부

```text
Concept: DI
- 정의: 객체가 직접 의존 객체를 만들지 않고 외부에서 주입받는 설계 방식
- 별칭: Dependency Injection, 의존성 주입

Concept: Bean
- 정의: Spring 컨테이너가 생성하고 관리하는 객체

Relation:
- DI → Bean: 기반 개념
- Bean → Bean Lifecycle: 확장 개념
- Controller ↔ Service: 맥락 개념
- Authentication ↔ Authorization: 대조 개념
```

### 5.2 자료 업로드 후 변화

```text
업로드 자료: "Spring DI와 Bean 정리.md"

AI 후보:
- DI
- Bean
- ApplicationContext

팀 검토:
- DI 승인
- Bean 승인
- ApplicationContext 반려

결과:
- DI 노드 활성화
- Bean 노드 활성화
- ApplicationContext 노드는 비활성 유지
```
